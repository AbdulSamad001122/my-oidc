Developer Guide

OIDC Provider Documentation

Welcome to the MyOIDC developer documentation. This guide explains every step required to integrate your application with our authentication server using the standard OpenID Connect (OIDC) Authorization Code flow.

1 Register Your Application

Before your application can communicate with MyOIDC and authenticate users, you need to introduce it to our system. This process provides you with unique credentials.

A
Go to the Dev Portal

Navigate to the Dev Portal and sign up or log in. This is your command center for managing applications.

B
Create a New Client

Click "Register App". You will be asked for an Application Name and a Redirect URI. The Redirect URI is the exact URL where MyOIDC will send the user back after they successfully log in. For example: https://yourapp.com/callback.

C
Save Your Credentials

Upon registration, you will receive a Client ID (public identifier for your app) and a Client Secret (a private password only your backend should know). Save the secret immediately, as it is only shown once.

2 Discovery and Configuration

OpenID Connect is designed to be plug and play. Our server provides a "Discovery Document" that automatically tells any OIDC library exactly where to find all required endpoints.

GET /.well-known/openid-configuration

When a library queries this endpoint, it receives a JSON file mapping out the server's capabilities:

{
  "issuer": "https://myoidc.example.com",
  "authorization_endpoint": "https://myoidc.example.com/o/authorize",
  "userinfo_endpoint": "https://myoidc.example.com/o/userinfo",
  "jwks_uri": "https://myoidc.example.com/.well-known/jwks.json"
}

3 Authorization Code Flow

This is the core process of logging a user in. It happens in two main parts: sending the user to log in, and exchanging a temporary code for real access tokens.

A. Redirect to Authorize Endpoint

When a user clicks "Log In" on your website, you must redirect their browser to our /o/authorize endpoint. You need to include specific parameters in the URL so we know who sent them.

GET /o/authorize 
GET /o/authorize?client_id=YOUR_CLIENT_ID
  &redirect_uri=YOUR_REDIRECT_URI
  &response_type=code
  &state=RANDOM_STRING
client_id The ID you received in Step 1.
redirect_uri Where we send the user back to. Must match what you registered exactly.
response_type=code Tells the server you want a temporary authorization code.
state A random string you generate. We send it back so you can prevent CSRF attacks.

What happens next: The user sees a login screen and an account chooser. Once they authenticate and grant consent, we redirect their browser back to your redirect_uri with a temporary code in the URL.

B. Exchange Code for Tokens

Your backend server receives the temporary code. Now, your server must make a secure backend to backend POST request to our /o/token endpoint to trade that code for real tokens. This is done from your backend so your Client Secret is never exposed to the browser.

POST /o/token

The Request

POST /o/token
Content-Type: application/json

{
  "grant_type": "authorization_code",
  "code": "AUTH_CODE_FROM_URL",
  "client_id": "YOUR_CLIENT_ID",
  "client_secret": "YOUR_CLIENT_SECRET",
  "redirect_uri": "YOUR_REDIRECT_URI"
}

The Response

If successful, we return a JSON object containing the tokens.

{
  "access_token": "random_access_string",
  "id_token": "eyJhbGci...", // Signed JWT containing user details
  "token_type": "Bearer",
  "expires_in": 3600
}

C. Fetch User Info (Optional)

While the id_token already contains basic user information, you can use the access_token to fetch the latest profile data directly from our userinfo endpoint.

GET /o/userinfo 
GET /o/userinfo
Authorization: Bearer YOUR_ACCESS_TOKEN

4 Testing with Postman

If you do not have an application built yet, you can easily test the entire OAuth 2.0 flow using Postman's built-in authorization helper.

1

Open a new request tab and go to the Authorization tab.

2

Set the Type dropdown to OAuth 2.0.

3

Set Grant Type to Authorization Code.

4

Fill in the required URLs and credentials:

Auth URL:
http://localhost:8000/o/authorize
Token URL:
http://localhost:8000/o/token
Client ID:
From Dev Portal
Client Secret:
From Dev Portal
5

Click Get New Access Token. Postman will open a browser window and run the flow for you!
